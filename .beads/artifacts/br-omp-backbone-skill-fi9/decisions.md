# Decisions: br-omp-backbone-skill-fi9

## Decision Log

| # | Decision | Rationale | Confidence |
|---|----------|-----------|------------|
| 1 | Commit s2s closeout artifacts as a `close:` commit, not a `chore:` commit | The bead is already closed in the DB. The `/close` command's Phase 4 pattern uses `close: $BEAD_ID`. Following this pattern reconciles the git history with the bead state, matching the existing `close:` commits for other beads (8f73a9c, f4816fa, 194c79a). A `chore:` prefix would misrepresent the nature of the change — this is a closeout, not a general chore. | High |
| 2 | Commit README.md change separately from the s2s closeout | The README.md change (adding `/npm-release` to the command table) is semantically unrelated to the s2s closeout. Mixing them in one commit violates single-responsibility. Use a `docs:` prefix for README, `close:` for s2s. | High |
| 3 | Delete the entire `.omp/scripts/` directory, not just the `__pycache__/` subdirectory | The directory contains only the dead `__pycache__/hydrate-memory.cpython-311.pyc` file. No other files exist in `.omp/scripts/`. The standalone hydration script was deleted and its logic inlined into `.omp/commands/init.md` as a heredoc. Keeping an empty directory serves no purpose. | High |
| 4 | Add `__pycache__/` to `.gitignore` rather than `.omp/scripts/__pycache__/` | The general `__pycache__/` pattern catches Python bytecode anywhere in the repo, not just under `.omp/scripts/`. This is the standard Python gitignore pattern and protects against future Python scripts leaking bytecode. The existing `.gitignore` already uses general patterns (`.bv/`, `.env`, `.worktree/`). | High |
| 5 | Push to origin/main directly, no feature branch or PR | This is a mechanical hygiene task on the `main` branch: committing existing untracked artifacts, deleting dead files, and pushing an already-created unpushed commit. No code changes, no review needed. The repo has no CI workflows to run. Creating a feature branch + PR for gitignore + artifact commits would be ceremony without value. | High |
| 6 | Do NOT delete `.omp/RULES.md` despite it appearing to duplicate AGENTS.md content | Investigation revealed `.omp/RULES.md` is a harness-loaded sticky rule (always-apply), injected into the system prompt's `<generic-rules>` block. Per OMP context-files documentation (`omp://context-files.md`), `.omp/RULES.md` is loaded as an always-apply rule at the native project location, re-attached near the current turn across long sessions. Deleting it would strip the 6 workflow rules from every agent's system prompt. The AGENTS.md "Guardrails" section is different, softer content — not a true duplicate. | High |
| 7 | Commit the untracked `.br_history/*.jsonl` files alongside the s2s review-report.md | The `.br_history/` directory is already tracked in git (72 tracked files confirmed via `git ls-files`). The 6 untracked files from the s2s session follow the same pattern — they are bead state snapshots that belong in the repo. Committing them as part of the `close:` commit keeps the bead state consistent with the closeout. | High |
