# Decisions: br-omp-backbone-skill-9tl

## Decision Log

| # | Decision | Rationale | Confidence |
|---|----------|-----------|------------|
| 1 | Hydrate memory files inline in init.md via Python 3 heredoc, not a standalone script | Matches the "commands + skills only, no scripts" philosophy. Python 3 is already required by br skill and init.md Phase 3. A 50-line heredoc is simpler to maintain than a separate file that needs its own discoverability and versioning. The init command already uses this pattern for Honcho .env config. | High |
| 2 | Insert hydration as Phase 2.5 (after br init, before Honcho config) | br init must succeed first (Phase 2 creates the `.beads/` directory). Honcho config (Phase 3) derives `HONCHO_WORKSPACE_ID` from the project name — running hydration first means Honcho gets the detected name rather than a raw directory name. Phase 3's existing `re.sub` pattern handles idempotent replacement of the managed block regardless of the workspace name value. | High |
| 3 | Detect via config file presence, not file extension counting | Counting `.ts` vs `.rs` files is fragile (build artifacts, vendored deps, fixtures). Checking for a single canonical config file at the repo root (`package.json`, `Cargo.toml`, `go.mod`, `pyproject.toml`) is deterministic and fast. First match wins — the project's primary language is the one with the package manager config. | High |
| 4 | Replace only unfilled placeholders; skip already-filled fields (idempotency) | Users will edit memory files after the first `/init`. The hydration script must not overwrite their work. By checking if the exact placeholder text (e.g., `<project-name>`) is still present before replacing, we guarantee idempotency. This is the same strategy used by init.md Phase 3 for Honcho .env blocks. | High |
| 5 | Leave human-judgment fields as `<!-- TODO: fill in -->` markers, not empty or with placeholder text | Empty fields teach agents nothing. Preserving `<One sentence...>` placeholder text is actively harmful — agents might treat it as an actual project goal. A `<!-- TODO -->` HTML comment is invisible in rendered Markdown but visible to humans editing the file. It's a standard convention for "fill this in later." | High |
| 6 | Preserve all template bootstrap gotchas and architecture decisions | The 12 gotchas in gotchas.md describe the template's own behavior (workflow gate bypass, bv requiring history, memory staleness, etc.). These are universally true for any project using this template. The 5 decisions in decisions.md document why the template works the way it does (br/bv over linear, commands+skills over scripts, .omp/ over .pi/). Deleting them would destroy institutional knowledge. Users add their own project-specific entries below the template sections. | High |
| 7 | Hydration targets only `.omp/memory/project/` files, not `.omp/templates/` | Templates are the upstream source of defaults. Hydration operates on the instantiated copies in `memory/project/`. If we modified templates, every future clone would get pre-hydrated defaults that might be wrong for them. The separation between templates (canonical defaults) and memory files (project-specific instances) must be preserved. | High |

## Rejected Alternatives

| # | Alternative | Why Rejected | Risk if Re-introduced |
|---|-------------|--------------|----------------------|
| 1 | Standalone `hydrate-memory.py` script in `.omp/scripts/` | Violates "no scripts" philosophy. Adds maintenance burden (script must be updated when memory file format changes). Script discoverability is poor — agents wouldn't know to look for it. | Bloat: a one-time init task becomes a permanent script that needs versioning, testing, and documentation |
| 2 | Hydrate at clone time via git hooks or post-clone script | Requires users to run a script outside OMP. The `/init` command is the single entry point — adding a pre-init step creates confusion. Also, git hooks don't transfer on clone (security restriction). | Two-step init process; users skip the hydration step; placeholders persist |
| 3 | Pure bash detection with grep/sed | Cannot parse `package.json` (JSON) or `Cargo.toml` (TOML) reliably. Bash regex on JSON breaks on key ordering, nested objects, or comments. Silent failures produce wrong tech-stack values that are worse than placeholders. | Incorrect language/tooling values in memory files; agents make wrong decisions |
| 4 | OMP TypeScript extension for hydration | Over-engineered for a one-time initialization task. Requires a TypeScript compilation step that may not be available on a fresh clone. Extensions are for tool behavior modification (workflow gate), not init-time file templating. | Build tooling dependency; extension loading overhead for a task that runs once |
| 5 | Hydrate only project.md, leave other files as-is | Leaves conventions.md and tech-stack.md with placeholder language rows. The problem is systemic across all 5 files — fixing one file while leaving the others broken is incomplete. | Most of the token waste is in conventions.md and tech-stack.md (multi-choice placeholders); partial fix doesn't help |
| 6 | Delete placeholder sections entirely instead of marking with TODOs | Removing the "Goal" and "Success Criteria" sections from project.md changes the document structure. Future agents expect these sections to exist (they're referenced in conventions.md's memory maintenance table). A TODO marker preserves the structure while clearly indicating it needs human attention. | Missing sections cause agent confusion; memory maintenance table references non-existent sections |

## Assumptions

| # | Assumption | Validation | Invalidation Impact |
|---|------------|------------|---------------------|
| 1 | Python 3 is available on all systems that run this template | The br skill already requires Python 3 for JSON parsing. init.md Phase 3 already uses Python 3 for .env config. If Python 3 is missing, the template already fails at Phase 3 — Phase 2.5 doesn't add a new dependency. | If Python 3 is unavailable, skip Phase 2.5 with a warning. Memory files stay as placeholders. This is a graceful degradation — the template still works, just without auto-hydration. |
| 2 | Project language is indicated by exactly one config file at repo root | Most projects have a single primary language. Monorepos with multiple languages would need manual editing regardless. The "first match wins" heuristic covers 95% of cases. | Wrong language in memory files. Mitigation: the human edits the files manually after /init. This is documented in the init.md report. |
| 3 | Users want template gotchas and decisions preserved | The gotchas describe the template itself (workflow gate, br/bv behavior, memory staleness). Every project cloned from this template needs them. The decisions explain why the template is structured as it is — removing them would leave future contributors wondering about design choices. | If users don't want template content, they can delete it manually after /init. The hydration phase doesn't lock the files. |
| 4 | The hydration script won't need updating when memory file format changes | Memory files follow a stable frontmatter + Markdown body pattern. Placeholder patterns are delimited by `<` and `>` which won't appear in real project content. If the format changes, the script's regex patterns need updating. | Hydration fails silently (no matches found) or replaces wrong text. Mitigation: the script only replaces exact placeholder patterns, so format changes that remove those patterns just result in no replacements — a safe no-op. |
| 5 | Honcho Phase 3's existing `re.sub` pattern handles the updated workspace name from Phase 2.5 | Phase 3 replaces the entire managed `# BEGIN OMP HONCHO ... # END OMP HONCHO` block. The `HONCHO_WORKSPACE_ID` value inside the block is generated fresh each run. Phase 2.5 provides a better project name that Phase 3 can use. But Phase 3 currently derives the workspace name from `root.name.lower()` in its own Python script — to benefit from Phase 2.5, Phase 3 must read the project name from `project.md` instead. | If Phase 3 doesn't read from project.md, the Honcho workspace uses the raw directory name (current behavior). This is still functional, just less clean. Phase 3 enhancement is a nice-to-have, not a blocker. |
