# Decisions: br-omp-backbone-skill-close-memory-audit-tkt

## Decision Log

| # | Decision | Rationale | Confidence |
|---|----------|-----------|------------|
| 1 | Insert Memory Audit as a new phase between Phase 1 (Read Review) and Phase 2 (Close) in close.md | Review must complete first (need confirmation bead is correct). Close must happen after (audit gates the final close). Placing between ensures we have full context before auditing and can still block close. Alternative considered: appending as Phase 7 after Report — rejected because bead would already be closed, making it a post-hoc audit rather than a gate. | High |
| 2 | Phase is named "Phase 1.5: Memory Audit" rather than renumbering all phases (e.g., making it Phase 2 and shifting everything) | "Phase 1.5" clearly signals this is an insertion point without changing the identity of well-known phases (Phase 2: Close, Phase 3: Check Queue Impact). Agents and users already know "Phase 2 = Close" — renumbering would break muscle memory. The `.5` convention has precedent in `/create` which uses Phase 6b for worktree. | Medium |
| 3 | Use a mandatory checklist (table per memory file) rather than an algorithmic drift detection script | Memory files use freeform markdown tables with varying structure. Any regex or script-based detection would have high false-positive/negative rates. An agent reasoning checklist is more robust — the agent reads actual content and reasons about semantic drift. User approval gate mitigates hallucination risk. | High |
| 4 | Require per-file user approval (yes/no/skip) rather than batch approve-all/reject-all | Per-file granularity prevents all-or-nothing dynamics: a wrong proposal can be rejected without losing good ones. The extra interaction cost is acceptable because drift is the exception, not the rule — most beads need zero updates. | High |
| 5 | Hard STOP if user rejects ALL proposals — bead remains open | If all proposals are rejected, either the agent hallucinated (audit is broken) or the user is refusing to update stale memory (contract violation). In both cases, proceeding to close would violate conventions.md. The STOP forces resolution: re-run /close after fixing, or manually update memory files. | High |
| 6 | Do NOT auto-approve or auto-apply any memory edits — every edit requires explicit user confirmation | Automated memory edits are dangerous. The agent could misunderstand what changed, hallucinate a gotcha, or propose a decision that wasn't made. User approval is the safety valve. Aligns with guardrail "Ask before destructive." | High |
| 7 | Memory audit is forward-looking only — no retrospective audit of already-closed beads in this bead | Retrospective audit is a separate concern with different scope (reading 8 closed beads' PRDs, checking current memory state). Adding it here would be scope creep. Future bead candidate if staleness is detected. | High |
| 8 | Do NOT update conventions.md Memory File Maintenance section in this bead | The conventions.md text already correctly describes WHAT happens ("agent checks... proposes updates"). This bead implements the HOW (close.md phase). Updating conventions.md would be a self-referential change — the convention didn't change, the implementation did. | Medium |

## Rejected Alternatives

| # | Alternative | Why Rejected | Risk if Re-introduced |
|---|-------------|--------------|----------------------|
| 1 | Separate `/audit-memory` command instead of embedding in `/close` | Cleaner separation of concerns, but structurally optional — can be skipped. The whole point is that memory audit must happen on every close, not "when the user remembers." | Memory audit becomes optional, conventions.md mandate is effectively void. Users forget or skip it. Memory drift accumulates. |
| 2 | Script-based drift detection (Python script that parses markdown tables and diffs against bead artifacts) | Too brittle. Memory files use freeform markdown, bead artifacts have no fixed schema for "what changed," and completion-evidence.json is sparse. Would produce high false-positive/negative rates. Also violates Decision #2 (no scripts). | Maintenance burden, platform dependency, fragile parsing. False positives erode trust in the audit. |
| 3 | Add memory audit to `/review` instead of `/close` | Catches drift earlier (before PR creation). But review is about code quality, not documentation. Adding memory audit couples concerns and makes review reports longer. Also, review is automated parallel agents — can't do interactive user approval. | Review reports become bloated with non-code findings. User approval can't happen in review context. Review's purpose is diluted. |
| 4 | Auto-apply memory updates without user approval (trust the agent) | Faster, less user interaction. But agents hallucinate. Wrong memory file content degrades ALL future sessions — the blast radius of a wrong auto-edit is huge. | Wrong conventions, decisions, or gotchas are injected into durable memory. Every subsequent agent session loads incorrect context. Catastrophic. |
| 5 | Batch approval ("approve all changes? yes/no") | Simpler interaction. But one wrong proposal forces rejection of all. User must either accept a bad update or lose good ones. | Good updates are lost when user must reject-all to avoid one bad proposal. Erodes trust in the audit. |
| 6 | Add memory audit to `/plan` (proactive, not reactive) | Plan-time is when the bead's scope is defined — ideal time to flag memory impact. But plans are provisional — the bead might not ship as planned. Close-time is definitive: the work is done, evidence is final. | False positives during planning (plan says "might change conventions" but implementation doesn't). Extra work during an already-heavy phase. |
| 7 | Only audit the specific memory files the PRD mentions (scope-based audit) | More efficient — only read files that are relevant. But the agent can't know which files are relevant without reading them. And subtle impacts (e.g., a code change that implies a gotcha) wouldn't be caught. | Missed drift when bead impacts a memory file not explicitly named in PRD. Undermines the "comprehensive" mandate. |
| 8 | Add memory-health scoring (quantitative staleness metric) to the audit | Would provide a numeric measure of memory freshness. But "staleness" is semantic, not quantitative — a file can be 6 months old but perfectly accurate, or 1 day old and wrong. A score would be misleading. | Users focus on the number, not the content. "Score is 85/100, good enough" when a critical gotcha is missing. Metric becomes a proxy for actual audit quality. |

## Assumptions

| # | Assumption | Validation | Invalidation Impact |
|---|------------|------------|---------------------|
| 1 | Agents executing `/close` are capable of reading 5 memory files + bead artifacts and reasoning about semantic drift | Test by running `/close` with a capable model (e.g., Claude, GPT-4, DeepSeek-V4) on a bead known to have memory impact | If agents can't reliably detect drift, the checklist must be made more specific (more concrete triggers) or the approval gate must be tightened (agent flags ALL files as "uncertain" and asks user to review each) |
| 2 | Memory files exist and are readable at close time | The prerequisites already check that bead artifacts exist; memory files are part of the repo and should always be present | If a memory file is missing, the phase handles it with an explicit error and user prompt (Edge Case EC-1). The bead can't close until resolved. |
| 3 | Most beads do NOT require memory file updates (drift is the exception) | Track over 10-20 beads: what percentage trigger at least one memory file proposal? Expected: <30% | If most beads trigger proposals, the user interaction cost is higher than estimated. Consider adding a "batch approve" option as an escape hatch. |
| 4 | User is present and responsive during `/close` | `/close` is the final phase — the user is actively wrapping up and can respond to prompts | If the user is AFK, the audit will block indefinitely. This is acceptable — closing a bead is a deliberate act, not a background operation. |
| 5 | The 100-line addition to close.md does not degrade agent comprehension of the command | The current close.md is 74 lines; the addition brings it to ~180 lines, still well under the 600-line density target for artifacts | If agents start skipping the audit phase because close.md is "too long," the phase may need to be extracted to a separate skill loaded during /close |
| 6 | Memory file sizes stay within targets after audit-driven updates | Most updates are small (a new table row, a sentence change), not structural rewrites. Target sizes: project.md ≤1KB, conventions.md ≤4KB, others ≤2KB (AGENTS.md lines 100-102) | If an update would push a file over target, the edge case handler (EC-8) offers trim/skip options. Structural size growth is a separate concern for a future bead. |
| 7 | No other workflow phase depends on the exact numbering of close.md phases | Phase numbering is internal to close.md — external references (e.g., AGENTS.md command reference) refer to phases by name, not number | If external tooling (e.g., workflow-gate.ts, orchestrator skill) references close.md phase numbers, those references must be updated. Current state: no such references exist. |
